01Parada 01 · Objetivo

Propósito y objetivo del proyecto

Implementar un sistema distribuido real para la Autoridad de Transporte Urbano de Lima y Callao (ATU).

Objetivo

Rastrear en tiempo real la posición de una flota de buses, almacenar el historial en base de datos y visualizarlo en un mapa web, garantizando que el servicio siga disponible aunque el servidor principal falle.

Pregunta orientadora

¿Qué pasa si el servidor que recibe el GPS de los buses se cae? ¿Cómo se protege el sistema de accesos no autorizados?

Resultado logrado

Servidor Java con instancia principal y de respaldo, balanceo automático con HAProxy, firewall nftables y dashboard Leaflet.js mostrando los buses en http://10.230.60.102:7070/dashboard.html.

ComponenteRol en el sistema
BusCliente.javaSimula la flota de buses (BUS-101 a BUS-606), enviando GPS cada 3 segundos.
ServidorATU.javaRecibe ubicaciones por TCP, las guarda en MariaDB y expone una API HTTP/JSON.
MariaDBPersiste el historial de ubicaciones en la tabla ubicaciones.
HAProxyBalancea tráfico entre el servidor principal y el de respaldo; failover automático.
nftablesFirewall con política DROP y whitelist de puertos.
dashboard.htmlMapa Leaflet.js que consume la API y muestra los buses en tiempo real.
02Parada 02 · Vista general

Arquitectura general del sistema

Los buses nunca hablan directo con la base de datos; todo el tráfico pasa por HAProxy antes de llegar al servidor activo.

Nota: los buses y el dashboard nunca apuntan directamente al servidor principal o de respaldo; siempre hablan con HAProxy, que decide a cuál instancia reenviar la conexión.
03Parada 03 · Fundamentos

Conceptos clave del sistema

Cuatro ideas sostienen el proyecto: comunicación TCP, API REST, balanceo/failover y firewall perimetral.

TCP multihilo (buses → servidor)

Cada bus abre un Socket contra el servidor y envía texto plano ID_BUS|lat|lon cada 3 segundos. ServidorATU usa un ExecutorService con pool de hilos para atender varios buses a la vez sin que uno bloquee a otro.

API HTTP/JSON (servidor → dashboard)

El mismo proceso Java expone com.sun.net.httpserver.HttpServer con las rutas /buses (últimas posiciones), /health (para HAProxy) y /stats (métricas del propio servidor).

Balanceo y failover (HAProxy)

HAProxy expone un único punto de entrada (:9000 TCP, :9001 HTTP) y reparte hacia el servidor principal o el de respaldo. Si el principal deja de responder al health check, HAProxy lo saca de la rotación automáticamente.

Firewall perimetral (nftables)

Política DROP por defecto: todo el tráfico entrante se bloquea salvo los puertos explícitamente permitidos (SSH, TCP/HTTP de ambas instancias, HAProxy y dashboard).

Atención concurrente de buses (TCP)

static void iniciarTCP() {
    try (ServerSocket ss = new ServerSocket(TCP_PORT)) {
        ExecutorService pool = Executors.newCachedThreadPool();
        while (true) {
            Socket cliente = ss.accept();
            pool.submit(() -> manejarCliente(cliente));
        }
    }
}

07Parada 07 · Perímetro

Seguridad: firewall con nftables

Política DROP por defecto: se bloquea todo el tráfico entrante salvo lo explícitamente permitido.

1

Configuración aplicada (/etc/nftables.conf)

table inet filter {
    chain input {
        type filter hook input priority filter; policy DROP;
        iif "lo" accept
        ct state established,related accept
        ip protocol icmp accept
        tcp dport 22 accept      # SSH administración
        tcp dport 5050 accept    # Servidor TCP principal
        tcp dport 8080 accept    # HTTP API principal
        tcp dport 9000 accept    # HAProxy TCP
        tcp dport 5051 accept    # Servidor TCP respaldo
        tcp dport 8081 accept    # HTTP API respaldo
        tcp dport 7070 accept    # Dashboard web
        log prefix "[NFT DROP] " drop
    }
    chain forward { policy drop; }
    chain output  { policy accept; }
}
2

Aplicar y habilitar

sudo cp nftables.conf /etc/nftables.conf
sudo systemctl enable nftables
sudo systemctl restart nftables
sudo nft list ruleset
Puerto¿Para qué?
22SSH (administración remota)
5050 / 5051Buses → Servidor TCP (principal / respaldo)
8080 / 8081Dashboard → Servidor HTTP (principal / respaldo)
9000 / 9001 / 9999HAProxy TCP / HTTP / panel de estadísticas

Por qué importa: si alguien externo intenta conectarse a un puerto no listado (por ejemplo 3306, MariaDB), el firewall lo descarta en silencio, sin siquiera responder.
08Parada 08 · Resiliencia

Alta disponibilidad con HAProxy

El problema a resolver: si el servidor principal se cae, ningún bus debería perder conexión.

1

Configuración (/etc/haproxy/haproxy.cfg)

Frontend TCP :9000 → backend round-robin: principal localhost:5050 (health check cada 2s) y respaldo localhost:5051 (solo si el principal cae).

Frontend HTTP :9001 → backend round-robin: principal localhost:8080 (health check GET /health) y respaldo localhost:8081.

Panel de estadísticas: http://10.230.60.102:9999/stats

2

Prueba de failover realizada

  1. Se terminó el proceso del servidor principal con kill <PID>.
  2. HAProxy detectó la caída vía health check (intervalo 2s, umbral 3 fallos).
  3. A los ~6 segundos, el tráfico se redirigió automáticamente al respaldo (:8081).
  4. El endpoint /buses siguió respondiendo sin interrupción, ahora desde el respaldo.
  5. Al restaurar el principal, HAProxy lo reincorporó a la rotación automáticamente.
[ANTES]   curl http://localhost:9001/buses  →  HTTP 200 (desde :8080)
[FALLA]   kill <PID servidor principal>
[ESPERA]  HAProxy detecta la caída (~6s)
[DESPUÉS] curl http://localhost:9001/buses  →  HTTP 200 (desde :8081 RESPALDO)
[RESULTADO] Servicio continuó sin interrupción ✓
09Parada 09 · Visualización

Dashboards web con Leaflet.js

Tres vistas sobre los mismos datos: mapa en vivo, panel técnico y consulta para el pasajero.

1. dashboard.html — Mapa en vivo

Cada 3 segundos llama a fetch('/buses') (vía HAProxy :9001) y mueve los marcadores sobre OpenStreetMap. Incluye trayectorias históricas (últimos 20 puntos) y lista lateral de buses activos.

2. dashboard2.html — Panel de control

Vista técnica para el administrador: KPIs de mensajes TCP recibidos, uptime, estado de cada nodo (buses, HAProxy, servidores, MariaDB) y log de eventos en tiempo real.

3. dashboard3.html — Consulta de viaje

Simulador para el pasajero: elige bus, paradero de origen/destino y condición de tráfico; estima tiempo de espera (con GPS real del bus) y tiempo de viaje, con mini-mapa de referencia.

Navegación: los tres dashboards están enlazados entre sí desde su barra superior, así que basta con abrir dashboard.html y moverse entre vistas con un clic.

Servido con:

python3 -m http.server 7070 --bind 0.0.0.0
# http://10.230.60.102:7070/dashboard.html

Comunicación

El servidor acepta múltiples clientes concurrentes con un ExecutorService, procesando el protocolo ID|lat|lon sin bloquear a otros buses.

Persistencia y API

MariaDB guarda cada ubicación con timestamp automático; el endpoint /buses devuelve la última posición por bus mediante GROUP BY.

Resiliencia y seguridad

HAProxy logra failover transparente en menos de 10 segundos; nftables reduce la superficie de ataque a solo los puertos necesarios.
